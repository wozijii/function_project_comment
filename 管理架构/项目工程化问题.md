# 项目工程化问题

> 入职第一周复盘 —— 工程化不是"把功能写出来"，而是"把功能做好、做稳、做长久"。

---

## 一、工程化核心问题清单

### 1. AB测试
- 新功能不要一把梭，用灰度/开关控制放量
- 关键指标要提前定好（转化率、报错率、响应时间）
- 能快速回滚，出问题秒切旧版本

### 2. 边界处理
- 空值、空数组、超长文本、极端数字 —— 都要兜住
- 列表为空时展示什么？文字超出怎么截断？
- 用户输入永远不可信，前后端都要校验

### 3. 兜底方案
- 接口挂了展示什么？（骨架屏 / 默认值 / 重试按钮）
- 图片加载失败要有占位图
- JS 报错不能白屏，至少给个友好提示

### 4. 稳定性（跑起来别挂）
- 核心链路要有监控和告警
- 关键操作加 try-catch，别让一个错误搞崩整个页面
- 内存泄漏、定时器未清除、事件未解绑 —— 这些隐性炸弹要排查

### 5. 兼容性
- 浏览器兼容：最低支持到哪个版本？
- 屏幕适配：手机、平板、PC 都要看
- 网络兼容：弱网、断网、切换网络时的表现
- 配合后端修改

### 6. 接口交互明确
- 前端传什么、后端返什么，字段类型、必填/选填，文档先写清楚
- 状态码规范：成功、失败、部分成功，各是什么码？
- 接口变更要通知，最好有版本控制

### 7. 复位测试
- 页面刷新后状态是否正确？
- 退出登录后缓存是否清干净？
- 多标签页之间数据是否冲突？

### 8. 压力测试
- 接口并发上限是多少？
- 大数据量下页面会不会卡？（1000条数据渲染、大表单提交）
- 提前摸底，别等上线后才崩

## 二、补充的工程化问题

### 9. 日志与监控
- 关键操作要有日志记录（谁、什么时间、做了什么）
- 前端错误上报（Sentry 等）
- 后端异常日志要完整，方便排查

### 10. 安全性
- XSS、CSRF、SQL注入 —— 基本防护不能少
- 敏感信息（密码、token）不存 localStorage，用 httpOnly cookie
- 权限校验后端做，前端只是展示控制

### 11. 可维护性
- 代码规范：ESLint / Prettier 统一风格
- 组件化：可复用的东西不要写两遍
- 注释：关键逻辑写清楚 WHY，不只是 WHAT

### 12. 发布流程
- 分支管理：feature → develop → release → main
- 上线前 checklist：测试通过？配置对了？回滚方案有？
- 灰度发布：先小范围验证，再全量放开

### 13. 性能
- 首屏加载时间、接口响应时间要有基线
- 图片压缩、代码分割、懒加载 —— 基本优化要做
- 不必要的请求要消除（重复请求、无效轮询）

### 14. 数据一致性
- 乐观更新 vs 等待响应 —— 选哪种要统一
- 缓存过期策略：多久刷新？手动刷新还是自动？
- 多端数据同步：改了之后其他设备能看到吗？

## 三、工程化流程（简版）

```
需求评审
  ↓
  ├─ 接口文档先行（前后端对齐）
  ├─ 明确边界情况和异常场景
  └─ 确定兼容性要求
  ↓
开发阶段
  ├─ 代码规范（ESLint + Prettier）
  ├─ 边界处理 + 兜底方案
  ├─ 关键逻辑加日志
  └─ 自测：正常 + 异常 + 边界
  ↓
联调阶段
  ├─ 接口交互验证
  ├─ 复位测试（刷新/重进/多标签）
  └─ 兼容性测试（多浏览器/多设备）
  ↓
测试阶段
  ├─ 功能测试
  ├─ 压力测试（大数据量/高并发）
  ├─ 安全测试（基本项）
  └─ 性能测试（首屏/关键路径）
  ↓
发布阶段
  ├─ 上线 checklist
  ├─ 灰度放量（AB测试）
  ├─ 监控告警就位
  └─ 回滚方案 ready
  ↓
上线后
  ├─ 关注监控和错误上报
  ├─ 收集用户反馈
  └─ 复盘 → 沉淀为规范
```

---

## 四、一句话总结

> **功能只是起点，工程化是让它在真实世界里活下去的能力。**
>
> 写代码时多想一步：如果这里出错了怎么办？如果数据量大了怎么办？如果用户环境不一样怎么办？
> 这三个问题想清楚了，工程质量就上去了。
