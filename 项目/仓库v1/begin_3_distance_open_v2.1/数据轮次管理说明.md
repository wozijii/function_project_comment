# 数据轮次管理修改说明

## 修改概述

本次修改实现了按照 `simulate_id`(轮次)进行数据隔离的机制,确保每个轮次之间相互独立,并在最后一个 push 时进行数据合并和生图。

## 核心逻辑

### 1. 目录结构

```
data/
├── latest_input.json              # 最新的输入数据(用于获取当前轮次信息)
├── data_185/                      # 轮次185的独立目录
│   ├── workers.json               # 该轮次的员工数据
│   ├── tasks.json                 # 该轮次的任务数据
│   ├── output_assignment.json     # 该轮次最新push的输出
│   ├── push_0_metadata.json       # push 0的元数据
│   ├── push_1_metadata.json       # push 1的元数据
│   ├── push_2_metadata.json       # push 2的元数据
│   ├── merge.json                 # 每次push的合并结果
│   ├── latest_push_index.json     # 最新的push索引
│   └── merged_all_outputs.json    # 当last_push=True时,合并所有push的输出
├── data_186/                      # 轮次186的独立目录
│   └── ...
└── archive/                       # 归档目录
```

### 2. 数据隔离机制

每个轮次(`simulate_id`)的数据存储在独立的目录 `data/data_{simulate_id}/` 中:
- 每个push会覆盖该轮次目录下的 `workers.json` 和 `tasks.json`
- 每个push会生成带时间戳的 `push_{push_index}_metadata.json` 文件
- 每个push会生成对应的 `output_assignment.json` 和 `merge.json`

### 3. last_push 判断逻辑

当 `last_push` 为 `True` 时:
1. **收集数据**: 收集该轮次所有push的输出结果
2. **合并输出**: 将所有push的 `output_assignment.json` 内容合并为 `merged_all_outputs.json`
3. **生图处理**: 调用 `deal_all_data_sendout.main()` 进行数据处理
4. **生成视频**: 调用 `view.view_nice_main` 生成可视化视频

## 工作流程

### 单个Push的处理流程

1. **接收请求**: `/process` 接口接收包含 `simulate_id`, `push_index`, `last_push` 的数据
2. **保存输入**: 保存到 `data/latest_input.json` 和归档目录
3. **处理数据**: `deal_data()` 创建轮次目录并保存 `workers.json`, `tasks.json`
4. **任务分配**: `main()` 读取轮次目录数据,生成 `output_assignment.json`
5. **发送回调**: `send_output_fast()` 合并数据并发送到回调接口
6. **判断结束**:
   - 如果 `last_push=False`: 仅发送回调,结束
   - 如果 `last_push=True`: 收集所有push数据,执行生图流程

### 多轮次并发处理

由于每个轮次使用独立目录:
- **轮次185的push 2** 和 **轮次186的push 0** 可以同时处理
- 它们读写不同的目录,互不影响
- 只有当某个轮次的 `last_push=True` 时才会生图

## 兼容性说明

- 保留了默认的 `/download/{fname}` 路由(读取 `data/` 根目录)
- 新增 `/download/{simulate_id}/{fname}` 路由(读取轮次目录)
- 如果无法获取 `simulate_id`,会回退到使用 `data/` 根目录

## 注意事项

1. **目录清理**: 随着轮次增加,`data/` 目录下会有很多 `data_{id}/` 子目录,需要定期清理旧数据
2. **生图逻辑**: 只有在 `last_push=True` 时才会调用生图模块,避免重复生图
3. **数据合并**: 由于每个push会覆盖 `workers.json` 和 `tasks.json`,最终生图使用的是最后一个push的输入数据
4. **回调机制**: 每个push都会发送回调,不仅仅是最后一个push

## 其他相关文件的修改

### 5. deal_all_data_sendout.py

**主要改动**:
- 从读取 `data/archive` 目录改为读取当前轮次的 `merged_all_outputs.json`
- 直接使用 `send_output_fast.py` 生成的合并数据,避免重复处理
- 输出文件保存到 `view/input_data_old/old_data.json` 用于生图
- 同时备份到轮次目录 `final_merged_output.json`

**关键逻辑**:
```python
# 读取 merged_all_outputs.json（包含该轮次所有 push 的数据）
merged_file = round_dir / "merged_all_outputs.json"

# 保存到 view 目录用于生图
output_file = output_dir / "old_data.json"
```

### 6. view/deal_nice_data.py

**主要改动**:
- 从读取固定的 `data/workers.json` 改为读取轮次目录下的文件
- 通过 `latest_input.json` 获取当前 `simulate_id`
- 动态构建轮次目录路径 `data/data_{simulate_id}/`
- 输出仍保存到 `view/input_data_nice/nice_data.json`

**关键逻辑**:
```python
# 获取当前轮次的 simulate_id
with open('../data/latest_input.json', 'r', encoding='utf-8') as f:
    latest_input = json.load(f)
simulate_id = int(latest_input.get('simulate_id', 0))

# 确定数据目录
data_dir = Path(f"../data/data_{simulate_id}")
workers_path = data_dir / "workers.json"
output_assignment_path = data_dir / "output_assignment.json"
```
