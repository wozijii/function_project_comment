## 一、项目概述

### 1.1 系统简介

这是一个基于 **FastAPI** 的仓库拣货任务智能分配与优化系统，通过先进的算法实现：

- ✅ **智能任务分配**：支持距离优先/负载均衡/混合策略
- ✅ **路径优化**：TSP算法优化SKU拣货顺序
- ✅ **路径规划**：A*算法计算实际可行走路径
- ✅ **可视化仿真**：生成员工移动轨迹的MP4视频
- ✅ **多轮次管理**：支持多轮次并发处理和数据隔离
- ✅ **实时监控**：状态追踪和进度反馈

### 1.2 技术栈

| 组件 | 技术 |
|------|------|
| Web框架 | FastAPI |
| 路径规划 | A*算法 |
| 优化算法 | TSP求解器 |
| 数据处理 | NumPy |
| 可视化 | Matplotlib + FFmpeg |
| API文档 | Swagger UI (自动生成) |

### 1.3 核心指标

| 指标 | 说明 |
|------|------|
| 仓库尺寸 | 66.33m × 46.9m (263×165 网格) |
| 移动速度 | 1.2 米/秒 |
| 拣货速度 | 按SKU配置（秒/包） |
| 视频帧率 | 10 fps (100ms/帧) |
| 仿真速度 | 3 步/帧 |

## 三、系统入口

### 3.1 主入口：app.py (FastAPI服务器)

**文件位置**：`[app.py](app.py)`

**作用**：提供HTTP API接口，是整个系统的Web服务入口

**核心接口**：

| 接口 | 方法 | 功能 |
|------|------|------|
| `/process` | POST | 任务分配接口（异步后台执行） |
| `/simulate` | POST | 仿真接口（生成可视化视频） |
| `/status` | GET | 获取系统运行状态 |
| `/status/process` | GET | 获取任务分配状态 |
| `/status/simulate` | GET | 获取仿真状态 |
| `/stop` | POST | 停止当前任务 |

**工作流程**：

```
前端请求 → app.py (接收数据)
    ├─→ validate_payload_data() (数据验证)
    ├─→ 保存输入数据到 data/latest_input.json + 归档
    ├─→ deal_data() (数据预处理，按simulate_id分目录)
    ├─→ main.main() (后台线程执行任务分配)
    └─→ 立即返回响应 (不阻塞，任务在后台执行)
```

**关键特性**：
- **异步执行**：使用 `ThreadPoolExecutor` 后台运行任务
- **数据验证**：检查前端传入数据的类型（坐标、时间、数量等）
- **日志系统**：按日期创建日志目录 `logs/YYYY-MM-DD/`
- **状态追踪**：实时更新任务进度，供前端轮询查询

**文件位置**：`[main.py](main.py)`

**作用**：实现完整的任务分配、优化、输出流程

### 4.2 任务分配详细流程

```python
# 在 TaskAllocator.allocate_task() 中

1. 判断任务类型
   ├─ task.priority > 0 ? (紧急任务)
   │   └─ YES → calculate_urgent_insertion()
   │       └─ 插入位置: 有执行中? → 后面 : 无执行中? → 最前面
   │
   └─ task.priority == 0 ? (普通任务)
       └─ YES → calculate_best_insertion()
           ├─ 候选位置:
           │   ├─ 任务数≤5: 全部位置
           │   └─ 任务数>5: 智能候选 (最接近任务的前后 + 两端)
           │
           └─ 对每个候选位置计算:
               ├─ optimize_sku_order_in_task() - TSP优化SKU顺序
               ├─ 计算移动时间 = distance / 1.2
               ├─ 计算拣货时间 = Σ(skuPickSpeed × inspectedQuantity)
               └─ 返回 (total_time, distance, end_grid)

2. 选择最优员工
   ├─ ALLOCATION_STRATEGY == 'distance_first' ?
   │   └─ YES → 选择 total_distance 最小的
   │
   ├─ ALLOCATION_STRATEGY == 'load_balance' ?
   │   └─ YES → 选择 total_time 最短的
   │
   └─ ALLOCATION_STRATEGY == 'hybrid' ?
       └─ YES → 选择 (distance + alpha × load_time) 最小的

3. 插入任务
   └─ update_worker_schedule(worker, task, position)
```

### 4.3 任务优化详细流程

```python
# 在 TaskAllocator.optimize_worker_tasks() 中

1. 分离任务
   ├─ urgent_tasks = [task for task in tasks if task.priority > 0]
   └─ normal_tasks = [task for task in tasks if task.priority == 0]

2. 紧急任务优化
   └─ 保持原有顺序 (不允许重排)

3. 普通任务优化
   └─ 按距离排序 (从当前位置到任务第一个SKU的距离)

4. 合并任务列表
   └─ final_tasks = urgent_tasks + normal_tasks

5. 优化每个任务的SKU顺序
   for each task in final_tasks:
       └─ TaskOptimizer.optimize_sku_order_in_task(task, start_grid)
           ├─ 1. SKU真实坐标 → batch_real_to_grid() → 网格坐标
           ├─ 2. snap_to_nearest_free() → 修正到可行走位置
           ├─ 3. 构建距离矩阵 (使用A*算法)
           └─ 4. TSPSolver.solve() → 返回最优路线和总距离
```

### 4.4 时间计算详细流程

```python
# 在 WarehouseOptimizationSystem._calculate_task_schedule() 中

for each worker:
    current_time = current_timestamp
    current_location = worker.current_location

    for each task in worker.allocated_tasks:
        # 跳过已完成的任务
        if task.task_status == "已完成":
            continue

        # 保持执行中原有预测时间
        if task.task_status == "执行中":
            current_time = task.predict_complete_time
            current_location = task.predict_complete_location
            continue

        # 计算新任务的时间
        ├─ 移动时间 = path_distance / MOVEMENT_SPEED
        ├─ 拣货时间 = Σ(sku.sku_pick_speed × sku.inspected_quantity)
        ├─ task.predict_start_time = current_time
        ├─ task.predict_complete_time = current_time + 移动时间 + 拣货时间
        └─ current_time = task.predict_complete_time
```

## 六、核心算法模块

### 6.1 TaskAllocator - 任务分配器

**文件位置**：`[core/task_allocator.py](core/task_allocator.py)`

**策略支持**：

```python
# config.py
ALLOCATION_STRATEGY = 'hybrid'  # 'distance_first' | 'load_balance' | 'hybrid'
HYBRID_ALPHA = 0.2  # 负载均衡系数
```

**策略说明**：

| 策略 | 选择标准 | 适用场景 |
|------|----------|----------|
| `distance_first` | 增量距离最短 | 最小化全局移动距离 |
| `load_balance` | 完成时间最短 | 最快完成所有任务 |
| `hybrid` | 移动时间 + α × 负载时间 | 平衡距离和负载 |

**关键方法**：

| 方法 | 功能 |
|------|------|
| `allocate_task()` | 分配任务给最合适的员工 |
| `calculate_best_insertion()` | 计算普通任务的最佳插入位置 |
| `calculate_urgent_insertion()` | 计算紧急任务的插入位置 |
| `optimize_worker_tasks()` | 优化员工的任务顺序 |

### 6.2 TaskOptimizer - 任务优化器

**文件位置**：`[core/task_optimizer.py](core/task_optimizer.py)`

**功能**：
- 使用 **TSP求解器** 优化单个任务内SKU的拣货顺序
- 计算任务完成时间和距离

**TSP优化流程**：

```
SKU真实坐标
    ↓
batch_real_to_grid()
    ↓
网格坐标 (col, row)
    ↓
snap_to_nearest_free()
    ↓
修正到可行走位置
    ↓
构建距离矩阵
    ↓
A*算法计算实际路径距离
    ↓
TSPSolver.solve()
    ↓
返回最优路线和总距离
```

**关键方法**：

| 方法 | 功能 |
|------|------|
| `optimize_sku_order_in_task()` | 优化SKU拣货顺序 |
| `calculate_task_completion_time()` | 计算任务完成时间和距离 |

### 6.3 PathFinder - A*路径规划

**文件位置**：`[core/path_finder.py](core/path_finder.py)`

**算法**：
- A*算法，曼哈顿距离作为启发函数
- 距离缓存机制避免重复计算

**输入**：起点 `(row, col)` 和终点 `(row, col)`
**输出**：实际米数（不是网格步数）

**关键方法**：

| 方法 | 功能 |
|------|------|
| `find_path()` | 返回完整路径 |
| `find_path_length()` | 返回路径距离（米） |

### 6.4 TSPSolver - TSP求解器

**文件位置**：`[core/tsp_solver.py](core/tsp_solver.py)`

**算法**：
- 暴力枚举或启发式算法
- 时间限制：5秒

**输入**：距离矩阵
**输出**：最优路线和总距离

    API_PORT = 8000                    # API端口

### 10.3 分配策略详解

#### 距离优先 (`distance_first`)

```python
# 选择增量距离最短的员工
metric = incremental_distance
```

**优点**：最小化全局移动距离
**缺点**：可能导致负载不均衡

#### 负载均衡 (`load_balance`)

```python
# 选择完成时间最短的员工
metric = completion_time
```

**优点**：任务完成时间更均衡
**缺点**：总移动距离可能增加

#### 混合策略 (`hybrid`)

```python
# 平衡距离和负载
metric = incremental_distance + alpha * worker_current_load
```

**优点**：兼顾距离和负载
**缺点**：需要调整α系数

## 附录：关键概念说明

### A. 坐标系统

**真实坐标**：(米) 如 `(66.33, 46.9)`

**网格坐标**：`(col, row)` 格式，从地图文件读取

**内部格式**：`(row, col)` 格式，用于A*算法

**转换工具**：
- `real_to_grid(x, y)`: 真实坐标 → 网格坐标 (col, row)
- `batch_real_to_grid(coords)`: 批量转换
- `snap_to_nearest_free(col, row)`: 修正到最近的可行走位置

### B. 任务状态

| 状态 | 说明 | 优化行为 |
|------|------|---------|
| 未开始 | 待执行的任务 | 参与优化 |
| 执行中 | 正在执行的任务 | 保持原有预测时间 |
| 已分配 | 已分配但未开始 | 参与优化 |
| 已完成 | 已完成的任务 | 跳过，不参与优化 |

### C. 任务优先级

| 优先级 | 说明 | 插入限制 |
|--------|------|---------|
| 0 | 普通任务 | 可插入任何位置 |
| >0 | 紧急任务 | 只能插入当前任务之后 |

### D. 常见问题

**Q1: 坐标格式混淆怎么办？**
A: 代码中混用了 `(col, row)` 和 `(row, col)` 格式，需要仔细检查。建议在注释中明确标注格式。

**Q2: 如何调试距离计算？**
A: 检查以下几点：
1. 坐标转换是否正确
2. 是否使用了 `snap_to_nearest_free` 修正坐标
3. 验证 `(row, col)` vs `(col, row)` 格式

**Q3: 紧急任务为什么总是插入到当前任务之后？**
A: 这是设计限制，紧急任务不允许打断当前正在执行的任务，只能插入到队列中当前任务之后。

**Q4: 如何提高任务分配速度？**
A: 当员工任务数>5时，系统会自动优化候选位置，只尝试最接近任务的前后位置和两端位置，而非全部位置。
