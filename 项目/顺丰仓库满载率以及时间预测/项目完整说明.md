# 笼车装载算法流式项目完整说明

本文档用于从 0 到 1 讲清楚当前项目：项目做什么、数据从哪里来、接口怎么调用、算法怎么处理、结果怎么输出、配置在哪里改、怎么启动、怎么排查问题。

## 1. 项目一句话说明

本项目是算法端服务，用来接收软件后端实时推送的多路摄像头图片帧和六面扫/运单信息，通过时间匹配后执行 SAM3/VLM 等算法处理，并把结果实时回调给软件后端。

当前项目已经从旧的离线 workflow：

```text
输入视频文件 + ref.txt -> 一次性跑完整流程 -> 生成 outputs 结果
```

改成实时流式服务：

```text
软件后端实时推图片帧
软件后端实时推六面扫/运单信息
算法端按时间匹配图片和运单
算法端后台异步处理
算法端回调两个 JSON 给软件后端
```

## 2. 三方业务关系

当前业务里有三个主体：

```text
顺丰侧
  提供真实业务数据来源，例如视频、六面扫、运单、格口、体积等信息

软件后端
  访问顺丰侧/业务系统
  获取视频或图片帧
  获取六面扫/运单信息
  整理成算法需要的接口格式
  推送给算法服务
  接收算法回调结果
  负责保存业务数据、查询 badcase、录像回溯等

算法端，也就是本项目
  接收图片帧
  接收六面扫/运单信息
  做时间匹配
  做 SAM3/VLM/笼车状态处理
  输出两个 JSON 给软件后端
```

整体关系：

```text
顺丰侧数据
  |
  | 视频/图片/六面扫/运单/体积/格口等
  v
软件后端服务器
  |
  | POST 图片帧 multipart/form-data
  | POST 六面扫 JSON
  v
算法服务 FastAPI
  |
  | 时间匹配 + SAM3 + VLM + 结果整理
  v
软件后端结果接收接口
  |
  | 保存结果 / 查询 badcase / 后续业务逻辑
  v
软件后端业务系统
```

## 3. 当前项目核心目标

当前项目目标不是离线生成一堆文件，而是作为一个长期运行的服务：

```text
服务一直启动
软件后端随时推图片
软件后端随时推六面扫信息
算法端匹配后处理
处理完成马上回调结果
处理过的数据从内存释放
```

测试阶段会保存输入图片、六面扫信息、回调 JSON，方便排查问题。生产阶段可以关闭输入图片保存。

## 10. 图片帧输入接口

### 10.1 接口地址

```text
POST /api/v1/frame-batches
```

Content-Type：

```text
multipart/form-data
```

注意：当前已经不接收 base64 JSON 图片。

### 10.2 字段说明

| 字段 | 是否必填 | 类型 | 说明 |
|---|---|---|---|
| `camera_id` | 是 | string，可重复 | 摄像头 ID，例如 `camera_1`、`camera_4` |
| `transmit_time` | 是 | string，可重复 | 图片时间，建议毫秒级，例如 `2026-07-10T08:40:45.301Z` |
| `image` | 是 | file，可重复 | 图片文件，支持 JPEG/PNG |
| `run_id` | 否 | string | 本次运行标识，默认 `default_run` |
| `batch_id` | 否 | string | 批次 ID，默认自动生成 |
| `batch_time` | 否 | string | 批次时间，默认取第一张图的 `transmit_time` |

### 10.3 字段数量要求

如果一次传 5 路摄像头，就需要：

```text
5 个 camera_id
5 个 transmit_time
5 个 image
```

三个字段数量必须一致。

### 10.4 调用示例

```bash
curl -X POST "http://10.10.95.229:9090/api/v1/frame-batches" \
  -F "camera_id=camera_1" \
  -F "transmit_time=2026-07-10T08:40:45.301Z" \
  -F "image=@/tmp/camera_1.jpg;type=image/jpeg" \
  -F "camera_id=camera_4" \
  -F "transmit_time=2026-07-10T08:40:45.029Z" \
  -F "image=@/tmp/camera_4.jpg;type=image/jpeg"
```

### 10.5 逻辑 JSON 形式

虽然实际传输是 multipart，不是 JSON，但逻辑上等价于：

```json
[
  {
    "camera_id": "camera_1",
    "transmit_time": "2026-07-10T08:40:45.301Z",
    "image": "camera_1.jpg"
  },
  {
    "camera_id": "camera_4",
    "transmit_time": "2026-07-10T08:40:45.029Z",
    "image": "camera_4.jpg"
  }
]
```

### 10.6 成功返回

接口接收成功后尽快返回 200。因为当前默认是异步处理，所以图片接口返回成功不代表算法结果已经生成，只代表图片已经接收并进入后台处理队列。

返回示例：

```json
{
  "status": "ok",
  "event": "frame_batch_received",
  "request_id": "xxx",
  "run_id": "default_run",
  "batch_id": "xxx",
  "frame_count": 2,
  "input_frame_count": 2,
  "camera_ids": ["camera_1", "camera_4"],
  "results_count": 0,
  "timings_ms": {
    "parse_multipart": 5.1,
    "validate_fields": 0.1,
    "read_upload_files": 0.2,
    "verify_images_and_build_items": 0.5,
    "close_upload_files": 0.0,
```

## 12. 时间格式要求

建议统一使用 ISO 格式，带毫秒：

```text
2026-07-10T08:40:45.301Z
```

也支持类似：

```text
2026-07-10 16:40:45.301
2026-07-10 16:40:45
2026-07-10T16:40:45.301
2026-07-10T16:40:45
```

不推荐：

```text
2026-07-10 16:40:45:301
```

这个最后一个毫秒前面是冒号，容易解析失败，应该改成点：

```text
2026-07-10 16:40:45.301
```

## 13. 图片和六面扫怎么匹配

当前匹配规则非常直接：

```text
abs(图片 transmit_time - 六面扫 loading_time) <= STREAM_MATCH_WINDOW_SEC
```

默认：

```bash
STREAM_MATCH_WINDOW_SEC=1.0
```

也就是图片时间和六面扫时间相差 1 秒以内，就认为可以匹配。

同时，当前算法要求一组结果至少匹配到这两个角色：

```text
cam1_placement
cam4_handoff
```

默认对应摄像头：

```text
camera_1 -> cam1_placement
camera_4 -> cam4_handoff
```

其他摄像头：

```text
camera_2 / camera_3 / camera_5 -> auxiliary_camera
```

辅助摄像头当前会接收、校验，测试阶段可以保存，但不进入当前主要算法处理。

## 14. 乱序处理怎么做

后端多线程推图片时，可能出现：

```text
第二帧先到
第一帧后到
```

当前服务不按请求到达顺序处理，而是按图片的 `transmit_time` 排序处理。

关键配置：

```bash
STREAM_REORDER_WINDOW_SEC=0.5
STREAM_WORKER_POLL_SEC=0.05
```

含义：

```text
STREAM_REORDER_WINDOW_SEC=0.5
  收到候选图片后先等 0.5 秒，让乱序图片有机会到齐，然后再按 transmit_time 选择最合适的一组。

STREAM_WORKER_POLL_SEC=0.05
  后台 worker 每 0.05 秒检查一次是否有可处理的数据。
```

这不会影响图片接口快速返回 200，因为当前默认：

```bash
STREAM_ASYNC_PROCESSING=1
```

图片接口收到并缓存后就返回，后续匹配、SAM3、VLM、回调在后台线程执行。

## 15. 当前算法处理流程

完整流程：

```text
1. 软件后端 POST 图片帧到 /api/v1/frame-batches
2. 接口检查 multipart 格式
3. 校验 camera_id / transmit_time / image 数量一致
4. 读取图片文件
5. 校验图片是 JPEG/PNG
6. 获取图片宽高、大小、时间
7. 保存输入图片到 outputs/images，测试阶段开启
8. camera_1/camera_4 进入内存缓存
9. 其他摄像头接收校验后释放，不进入算法主流程
10. 软件后端 POST 六面扫到 /api/v1/loading-manifests
11. 校验 loading_time / waybill / target_cage / chute_id / parcel_volume
12. 保存六面扫到 outputs/manifests，测试阶段开启
13. 后台 worker 按时间寻找匹配图片和六面扫
14. 匹配到 camera_1 + camera_4 后构造一组算法输入
15. 执行 SAM3
16. 执行 VLM 满载率识别
17. 生成 cargo_frame_result
18. 生成 cage_state_result
19. 保存两个回调 JSON 到 outputs/results
20. POST 两个 JSON 到软件后端接口
21. 使用完的图片 bytes 从内存释放
```

流程图：

```text
软件后端图片推送
  |
  v
/api/v1/frame-batches
  |
  | 校验 multipart / 图片 / 时间
  v
FrameItem 内存缓存
  |
  | camera_1 + camera_4 保留
  | 其他摄像头接收校验后释放
  v
等待六面扫匹配

软件后端六面扫推送
  |
  v
/api/v1/loading-manifests
  |
  | 校验运单/时间/笼车/格口/体积
  v
ManifestItem 内存缓存
  |
  v
按时间窗口匹配
  |
  v
SAM3 + VLM
  |
  v
两个输出 JSON
  |
  +--> cargo_frame_result -> 软件后端 actual-cart
  |
  +--> cage_state_result  -> 软件后端 fullness
```

## 16. 当前 SAM3 做什么

SAM3 当前封装在：

```text
workflow/realtime_sam3_engine.py
```

当前逻辑：

```text
收到图片 bytes
PIL 解码成 RGB
加载 SAM3 模型
使用文本 prompt: cargo
执行单帧检测
返回 detections
```

SAM3 模型路径：

```bash
SAM3_MODEL_PATH=/home/nvidia/cage_car/sam3/sam3_model
```

设备：

```bash
SAM3_DEVICE=cuda:0
```

当前 SAM3 是 lazy load：

```text
第一次真正用到 SAM3 时加载模型
后续复用同一个全局 engine
```

## 17. 当前 VLM 做什么

VLM 当前封装在：

```text
workflow/realtime_vlm_engine.py
```

当前逻辑：

```text
收到图片 bytes
PIL 解码
如果图片太大，缩放到最长边不超过 1024
重新编码 JPEG
转成 base64 image_url
请求本地 vLLM OpenAI-compatible 接口
让模型评估指定笼车满载率
从回答中解析 0-100 的数字
```

默认请求地址：

```text
http://127.0.0.1:8000/v1/chat/completions
```

默认模型名：

```text
Qwen/Qwen3-VL-8B-Instruct
```

输出的是满载率：

```text
fullness_percent
```

体积不再由 VLM 估算，体积直接来自六面扫字段：

```text
parcel_volume
```

## 19. 当前分辨率说明

当前真实接口可以接收任意图片尺寸，只要是合法 JPEG/PNG。

但是标定参数是按：

```text
960x540
```

建立的。

因此：

```text
960x540      最直接
1920x1080    可以按比例适配
2560x1440    也是 16:9，可以按比例适配
```

前提：

```text
画面没有裁剪变化
摄像头位置没有变化
```

## 26. 错误返回格式

接口错误会统一返回：

```json
{
  "status": "error",
  "message": "错误描述",
  "request_id": "xxx",
  "error": {
    "stage": "metadata_parse",
    "code": "missing_required_field",
    "message": "image is required",
    "status_code": 400,
    "field": "image"
  }
}
```

常见错误：

| HTTP 状态 | 原因 |
|---|---|
| 400 | 必填字段缺失、字段数量不一致、时间/数字格式错误、图片无效 |
| 415 | 图片接口不是 multipart/form-data |
| 410 | 旧单帧接口 `/api/v1/frames` 已禁用 |
| 422 | FastAPI 参数校验失败 |
| 500 | 未捕获内部错误 |

## 27. 为什么图片接口 200 但没有结果

图片接口返回 200 只代表图片接收成功，不代表已经匹配出结果。

没有结果常见原因：

```text
1. 没有同时收到 camera_1 和 camera_4
2. 图片 transmit_time 和六面扫 loading_time 超过匹配窗口
3. 时间格式解析失败
4. 六面扫缺字段，例如 waybill、target_cage、chute_id、parcel_volume
5. 后台 worker 还在等待乱序窗口
6. vLLM 没启动，导致 ready 不通过或 VLM 失败
7. 回调地址不可达，结果保存了但回调失败进入 pending
```

排查顺序：

```text
1. 看 /api/v1/ready
2. 看 /health 里的 buffered_frames / buffered_manifests / total_results
3. 看 outputs/frame_receiver/_service/service_runtime.log
4. 看 outputs/frame_receiver/_service/receiver_access.jsonl
5. 看 outputs/results/ 是否保存了回调 JSON
6. 看软件后端回调接口日志
```

## 30. 联调时软件后端应该怎么推


## 32. 重要结论

当前项目要记住这几件事：

```text
1. 这是实时服务，不是旧的一次性跑视频 workflow。
2. 图片从 /api/v1/frame-batches 进来。
3. 六面扫从 /api/v1/loading-manifests 进来。
4. 图片和六面扫按时间匹配。
5. 当前主要用 camera_1 和 camera_4。
6. 算法输出两个 JSON：cargo_frame_result 和 cage_state_result。
7. 输出 JSON 会回调给软件后端，也会保存到 outputs/results。
8. 体积来自后端 parcel_volume，不再算法估算。
9. 测试阶段输入图片和六面扫会保存，方便排查。
10. 大部分运行配置集中在 scripts/streaming/start_algorithm_service.sh。
```
